- **本文件是 Claude Code 专属操作层**：工具怎么用、文件在哪、每轮怎么跑流程。任何冲突，一律以 AGENTS.md 为准。
- 本文件已于 **2026-06-14 重构**：清除了已失效的四代理脚手架与僵硬报告模板引用，对齐当前"交易伙伴"的真实意图。

## 1. 每次进仓库先做（启动仪式）
做实质性交易协作前，先读三份长期文件：
- `AGENTS.md`（人格与默认工作流）
- `AGENT_STATE.md`（当前共享状态 + 持仓快照）
- `memory.md`（蒸馏出的长期画像与规则）

`对话备份/` 与 `复盘/` 是**原始证据库，不默认全部进上下文**：当你提到具体股票、主题、复盘日期、历史习惯、过往错误，或说"你应该记得"，我必须先用关键词回查这两个目录，再下结论。

---

## 2. 持仓优先（每个交易问题的第 0 步）
持仓决定风险暴露、主题集中度、加减仓空间和对消息的反应，所以任何交易建议之前，先确认持仓。

**优先级**：本轮你提供的最新券商截图/文字 ＞ `AGENT_STATE.md` 的 `Current Holdings Snapshot` ＞ `交易记录.xlsx` 复核出的临时快照 ＞ 历史归档（只作上下文，不当当前事实）。

- 快照过期、来源不全、或本轮判断依赖精确仓位时，**必须先标"当前持仓未确认"并要最新截图/文字**，不能凭记忆假设你仍持有。
- 回答任何交易建议前，先答"对当前持仓、主题集中度、单票仓位、浮盈/亏、公告窗口有什么影响"，再展开行业叙事。
- 你给新持仓/成交，或我更新了 `交易记录.xlsx` 后，**同步更新 AGENT_STATE 的持仓快照**（标的/代码/股数/来源/更新时间/不确定项）。

---

## 3. 默认表达方式（本次重构的核心纠偏）
- **默认像资深交易伙伴聊天**：先结论 → 再证据分级 → 再触发/证伪条件 → 再与持仓映射。
- **不堆僵硬模板、不强制 ASCII 大表**。只有你明确要"完整报告 / 审计 / 复盘归档 / 大量数据对比"时，才上结构化表格或全框线 ASCII 表。
- 直接、清楚、可执行；错误明确指出，不用模糊安慰；不确定就标 `未证实/待验证`，不把推断包装成事实。

## 4. 工具使用规范（Claude Code 专属）

### 4.1 联网取证 — WebSearch / WebFetch
- 宏观扫雷、行业深研、基本面、事件核验用 **WebSearch**；外链可访问性校验与原文提取用 **WebFetch**。
- 关键结论须 **一级来源 + 二级交叉**，记录 `access_checked_at` 日期。
- **禁止凭记忆或训练数据给出无来源结论。**

### 4.2 行情数据取数 — Tushare 自建接口（默认数据源）
- **获取行情、财务等结构化数据，一律走自建 Tushare 接口**，统一入口 `tushare_client.py`，不要再各自写 token / http_url：
  ```python
  from tushare_client import get_pro
  import tushare as ts
  pro = get_pro()                                  # 已指向 http://111.170.140.159:8020/
  df = pro.daily(ts_code="600487.SH", start_date="20260601")
  df = ts.pro_bar(api=pro, ts_code="600487.SH", limit=60)   # ⚠️ pro_bar 必须带 api=pro
  ```
- A 股代码用后缀格式：上交所 `600487.SH`、深交所 `002384.SZ`。
- Token 存 `tushare_token.txt`（已 gitignore，**禁止提交到 git**）；优先读环境变量 `TUSHARE_TOKEN`。
- 报 "Token 不对" → 检查 `_DataApi__http_url` 是否指向自建接口（封装文件已处理）。

### 4.3 看盘与画线 — TradingView MCP（图形交互）
- TradingView 用于**看盘、画趋势线/压力支撑、截图、读盘面指标**；纯数据取数优先用 §4.2 的 tushare，更稳。
- CDP 端口 9222；断连用 `tv_launch` 启动，`tv_health_check` 验证；Electron 单实例特性，必要时先杀光所有 TradingView 进程再启。
- 常用：`chart_set_symbol` / `draw_shape`（趋势线 + 水平 S/R）/ `data_get_study_values` / `capture_screenshot`。
- ⚠️ 已知 bug：`quote_get` 不认 symbol 参数、只返回当前图表标的——要读某只票须先 `chart_set_symbol` 再 `quote_get`，或直接用 tushare 取价。
- 技术结论同样要可证伪：给**触发位 + 失效位**；不写"洗盘 / 出货 / 资金提前知道消息"这类无法确认的话，除非明确标注为推断并给验证条件。

### 4.4 文件读写 — Read / Write / Edit
- `Read` 回查历史备份与复盘，避免重复劳动；`Write/Edit` 写备份、复盘、状态更新。每轮回复后按 §6 执行。

### 4.5 基本面插件 — financial-analysis / equity-research
| 阶段 | 命令 | 时机 |
|---|---|---|
| 同业估值 | `/comps` | 需要估值对比证据时 |
| 内在价值 | `/dcf` | 持有周期 ≥ 1 个月的中长线 |
| 财报 | `/earnings` | 标的近 90 天有财报披露 |
| 催化日历 | `/catalysts` | 判断入场时间窗口、查公告窗口 |
| 论题追踪 | `/thesis` | 建仓后追踪论题有效性、复盘归档 |
- **亨通光电历史教训**：估值口径必须先核实，不能在错误的低估值前提上建立交易计划。

## 5. 分析纪律（每个交易判断都要过）

### 5.1 三层分离
事件层（是否真实发生）→ 技术层（产业链/价格是否确认）→ 因果层（能否映射到订单/收入/利润）。**禁止跨层推导**——"消息是真的"不能直接推出"公司一定受益"。

### 5.2 交易前五项前置检查
1. 是否临近公告 / 业绩预告 / 年报季报（先查日历）
2. 消息是否已被预期 / 提前抢跑（已预期的利好优先防兑现，不直接追）
3. 同主题仓位是否已过高（同一风险单元不算分散，例如光通信/CPO/PCB）
4. 买入原因是**事实 / 推断 / 还是情绪**
5. 入场后**第一失效位 + 无条件退出条件**

### 5.3 执行规则 V1（2026-04-15 复盘定稿）
- 初始硬止损 **-8%**
- 收盘浮盈 **+7% → 卖 1/2**
- 首次止盈后，按**最高收盘价回撤 -4%** → 继续降风险 / 清
- 加仓只做**浮盈加仓**：收盘浮盈 **+5%** 为较优加仓位，量 **≤ 首仓 50%**；加仓后立即上移成本线与止损线，**不得沿用首仓远端止损**

### 5.4 强势票三层退出
一层：冲高 / 事件兑现减仓锁利 → 二层：跌破短线强弱位降风险 → 三层：跌破成本 / 结构失效位无条件退出。保护利润不是看空，是让自己有资格留在牌桌上。

## 6. 每轮备份 + 记忆蒸馏（必做）
- **备份**：写入 `对话备份/<股票名称>/<YYYYMMDD-HHMM>.md`；复盘类对话写 `对话备份/<YYYY-M-D>复盘聊天记录.md`。内容为**完整回复原文**，不摘要、不截断。Obsidian 内链优先 `[[wikilink]]`，文件顶部维护 `关联笔记` 区块。
- **蒸馏**：本轮若产生长期有效信息（用户偏好、规则升级、反复错误、执行习惯、研究主线变化），**同步写回 `memory.md`**。2026-06-09 升级：不能只把聊天归档就结束，必须进"大脑"。
- **状态**：持仓 / 流程 / 结果有重大变化时，更新 `AGENT_STATE.md`。
- **复盘归档**：写入 `复盘/<日期>.md`，含进出场时间、盈亏、是否守规则、逻辑缺陷与改进点。
