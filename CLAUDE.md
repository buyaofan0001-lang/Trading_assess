## 二、Claude Code 专属工具使用规范

### 2.1 联网检索（必须启用）
- 使用 **WebSearch** 工具执行宏观扫雷、行业深研、基本面信息检索。
- 使用 **WebFetch** 工具验证外链可访问性并提取原文（来源校验必做）。
- 每次引用前必须执行可访问性校验，记录 `access_checked_at` 日期。
- 禁止凭记忆或训练数据给出无来源结论。

### 2.2 文件操作
- 使用 **Read** 读取历史对话备份与复盘记录，避免重复工作。
- 使用 **Write/Edit** 写入对话备份与复盘文件。
- 对话备份必须在每轮回复完成后立即执行（见 AGENTS.md §16）。

### 2.3 代码执行（验证报告完整性）
- 使用 **Bash** 执行 `orchestrate_report.py` 做结构化输出校验：
  ```bash
  python3 subagents/orchestrate_report.py \
    --plan plan.json \
    --macro macro_industry.json \
    --bear bear_case.json \
    --technical technical.json \
    --output outputs/<股票名称>-<YYYYMMDD>.md
  ```

## 三、四代理协同编排（Claude Code 实现）

### 3.1 触发条件
用户提交交易计划后，Claude 作为 **总控守门官** 主控流程，通过 **Agent 工具** 并行派发三个子任务。

### 3.2 编排顺序

```
用户提交交易计划
      │
      ▼
[总控守门官] 校验输入 → 拆分任务
      │
      ├──────────────────────┐
      ▼                      ▼
[宏观行业研判官]        [空头对抗审计官]
 (general-purpose)      (general-purpose)
      │                      │
      └──────────┬───────────┘
                 ▼
         [技术执行判定官]
          (general-purpose)
                 │
                 ▼
         [总控守门官] 汇总 → 三段式报告
```

### 3.3 并行派发规则
- `宏观行业研判官` 与 `空头对抗审计官` 可**并行**执行（均依赖用户输入，互不依赖）。
- `技术执行判定官` 可与上述两者**并行**（依赖行情数据，不依赖研判结论）。
- 三个子代理完成后，**总控守门官**串行汇总、校验并输出最终报告。

### 3.4 各代理提示词文件
| 代理 | 提示词文件 |
|---|---|
| 总控守门官 | `subagents/总控守门官.md` |
| 宏观行业研判官 | `subagents/宏观行业研判官.md` |
| 空头对抗审计官 | `subagents/空头对抗审计官.md` |
| 技术执行判定官 | `subagents/技术执行判定官.md` |

## 四、输入校验清单（总控守门官必做）

在启动代理前，验证用户输入包含：

- [ ] 标的代码 + 名称 + 市场
- [ ] 入场逻辑（文字描述）
- [ ] 预期持有周期
- [ ] 关键催化
- [ ] 交易类型（剥头皮 / 短期 / 中期 / 长期）
- [ ] 联网检索已启用（WebSearch 可用）

缺少任意项 → 暂停并向用户补充，不得推进流程。

---

## 五、输出质量门槛（见 AGENTS.md §19.3）

| 维度 | 最低要求 |
|---|---|
| 宏观风险条数 | ≥ 8 |
| 宏观利好条数 | ≥ 8 |
| 基本面分析维度 | = 5，每维含结论+来源+日期+口径 |
| 空头利空条数 | ≥ 8，覆盖财务/行业景气/政策/公司公告 |
| 来源级别 | 关键结论须一级 + 二级双源交叉 |

任一不达标 → 补证或标记 `未证实/待验证`，不得降标输出。

---

## 六、禁止事项（摘自 AGENTS.md §14）

- 禁止做空、融券、反向交易建议。
- 禁止无来源结论。
- 禁止跨层推导（事件层 → 技术层 → 因果层，必须逐层验证）。
- 禁止忽略明显利空信号。
- 任一维度（逻辑或技术）验证不成立 → 交易计划否定，不得给出入场建议。

## 八、Financial Services Plugins 整合规范

已安装插件（需 **重启 Claude Code** 后生效）：
- `financial-analysis@financial-services-plugins`（核心，必装）
- `equity-research@financial-services-plugins`（研究工具集）

### 8.1 插件命令与研究流程的映射

| 研究阶段 | 插件命令 | 触发时机 |
|---|---|---|
| 基本面分析 → 估值与预期 | `/comps [标的]` | 宏观行业研判官进行同业估值对比时 |
| 基本面分析 → 财务建模 | `/dcf [标的]` | 需要内在价值测算时（中长期持仓） |
| 基本面分析 → 催化财务映射 | `/earnings [标的] [季度]` | 财报季后进行盈亏比较分析时 |
| 技术执行 → 催化剂日历 | `/catalysts [标的]` | 技术执行判定官定位时间窗口时 |
| 逻辑验证 → 论题追踪 | `/thesis [标的]` | 建仓后持续追踪交易论题有效性时 |
| 空头对抗 → 竞争分析 | `/competitive-analysis [标的]` | 评估竞争格局利空时 |

### 8.2 各代理使用插件的规则

**宏观行业研判官**：
- 估值与预期维度 → 必须调用 `/comps` 生成同业对比表格，作为估值结论的证据支撑。
- 若标的近90天内有财报披露 → 调用 `/earnings` 获取官方披露数据，作为利润质量与催化财务映射的一级证据。
- DCF 适用于中期以上（持有周期 ≥1个月）的交易计划。

**技术执行判定官**：
- 催化剂日历 → 调用 `/catalysts` 识别近期高影响催化剂，辅助判断入场时间窗口。
- 在技术面输出中明确注明最近一个催化剂日期及潜在影响。

**总控守门官（事后复盘阶段）**：
- 交易结束后，调用 `/thesis [标的]` 更新并归档论题评分卡（Thesis Scorecard）。
- 论题评分卡存储至：`对话备份/<股票名称>.md` 的对应轮次记录中。

### 8.3 插件数据来源优先级（遵循 financial-analysis 规范）
1. MCP 数据源（若可用）：S&P Kensho、FactSet、Daloopa
2. 官方 SEC EDGAR / 巨潮公告 直链
3. Bloomberg / FactSet 终端数据
4. 主流媒体一手报道
5. 社媒（一律标记 `未证实/待验证`）
